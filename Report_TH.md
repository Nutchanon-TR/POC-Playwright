# รายงานตรวจสอบและแก้ไขโค้ด (Code Review Report)

## ส่วนที่พบความผิดปกติและแนวทางแก้ไข (Code Issues & Fixes)

### 1. ปัญหาหยุดรันค้างที่ `dialog.waitFor({ state: 'visible', timeout: 3000 })`
**ตำแหน่งที่พบ:** `tests/support/helper/common/dialog.helper.ts` (ในฟังก์ชัน `confirmVisibleDialog`)
**สาเหตุ:** 
คำสั่ง `page.getByRole('dialog')` อาจจะเจอกล่อง Dialog มากกว่า 1 ตัวซ้อนกันในหน้าเว็บ (เช่น โค้ดยังไม่เคลียร์ Dialog ตัวเก่าทิ้ง หรือมี Backdrop บัง) ทำให้การเรียก `.waitFor()` แจ้งเตือนข้อผิดพลาดประเภท Strict mode violation ทันที ซึ่งข้อผิดพลาดแบบนี้จะทะลุ `.catch()` ออกมาเพราะมันเป็น Error จากระบบค้นหาของ Playwright เอง ไม่ใช่การ Timeout แต่อย่างใด
**วิธีแก้ไข (ดำเนินการแล้ว):**
เติม `.first()` หรือแคบ Locator ลงเพื่อให้ Playwright สนใจเฉพาะ Dialog ตัวบนสุดเสมอ:
```typescript
const dialog = page.getByRole('dialog').first();
await dialog.waitFor({ state: 'visible', timeout: 3000 }).catch(() => { });
```

### 2. ปัญหาหา Row ไม่เจอจน Timeout `await expect(rows.first()).toBeVisible({ timeout: 15000 });`
**ตำแหน่งที่พบ:** `tests/support/helper/corporate-report/corporate-profile.helper.ts` (ในฟังก์ชัน `editCorporateProfile`)
**สาเหตุ:**
ในโค้ดมีการคอมเมนต์บรรทัด `// await searchCorporateProfile(page, options.corporateId);` เอาไว้ และเลือกใช้วิธีกระโดดไปหน้าสุดท้ายของตาราง (`await gotoLastPaginationPage(page);`) แทนการใช้ Search Filter
เมื่อตารางมีการอัปเดตหรือโหลดข้อมูลไม่ทัน ข้อมูลของเทสที่ต้องการอาจจะไม่ได้อยู่ในหน้าสุดท้าย (บางทีอาจไปโผล่หน้า 1) ทำให้ฟังก์ชันไล่ควานหาในตารางไม่เจอ และไปติดอยู่ที่คำสั่ง `expect(rows.first()).toBeVisible()` จนกว่าจะหมดเวลา 15 วินาที
**วิธีแก้ไข (ดำเนินการแล้ว):**
การเขียน E2E Test ที่ดีเพื่อให้มั่นคงและเร็วที่สุด ไม่ควรใช้วิธีคลิกเปลี่ยนหน้าตาราง (Pagination) ลบทีละหน้าเพื่อควานหาข้อมูล แต่ให้ปลดคอมเมนต์ฟังก์ชัน Search เพื่อให้ระบบค้นหาตารางแบบเจาะจง `corporateId` จนเจอข้อมูลชัวร์ๆ ก่อน แล้วค่อยกด Edit ซึ่งตรงกับ Step ในไฟล์ `script.txt` 

### 3. โค้ดแปลกๆ ใน `table.helper.ts` (ข้อสังเกตเพิ่มเติม)
**ตำแหน่งที่พบ:** ในฟังก์ชัน `findTableRowByTexts` 
**การทำงานปัจจุบัน:** จะรันหาข้อมูลบนตารางหน้าปัจจุบันเสร็จแล้ว ถ้าไม่เจอจะไปกดปุ่ม "หน้าก่อนหน้า" (Previous page) หาต่อ
**ความผิดปกติ:** 
ตรรกะการไล่ควานหาข้อมูลจากหน้าสุดท้ายย้อนกลับมาหน้าแรกเรื่อยๆ เป็นวิธีรันลูปที่ทำให้เทสพัง (Flaky) ได้ง่าย และกินระยะเวลาในการรันนานมากเมื่อข้อมูลในตารางปริมาณเยอะขึ้น
**ข้อแนะนำในการเขียนในอนาคต:** 
ควรใช้กระบวนการ Filter / Search ผ่านหน้า UI แบบในฟังก์ชัน `searchCorporateProfile` แทนการกดเปลี่ยนทีละหน้าผ่าน Pagination เสมอ เพื่อย่นข้อมูลแถวในตารางให้เหลือเฉพาะ Record ที่เกี่ยวข้องกับ Test Data การทดสอบจริงๆ เท่านั้น
